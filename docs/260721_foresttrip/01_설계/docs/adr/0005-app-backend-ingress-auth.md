# 앱↔백엔드는 아웃바운드 터널 + Bearer 토큰 + TLS로 연결한다

- 상태: 제안 (2026-07-22 아키텍처 리뷰에서 도출)

백엔드는 **집(원격 Windows) PC**, 클라이언트는 **모바일 앱**이다. 알림받고 밖에서
원탭 핸드오프하는 게 제품 핵심이므로 폰은 **거의 항상 그 PC의 LAN 밖**에 있다. FCM은
아웃바운드 푸시(구글→폰)만 해결하고, `/v1/search`·`/v1/watches`·`/v1/auth/forest-login`
같은 REST는 **폰→집PC 인바운드**다. 따라서 API계약 §0의 *"개인 MVP는 단순 토큰/무인증도
허용(내부망 전제)"* 는 실제 토폴로지와 모순이며, 인그레스·인증·TLS를 확정하지 않으면 착수할
수 없다.

**위협:**
- 무인증 인터넷 노출 시 엔드포인트를 찾은 누구나 **단일 숲나들e 계정을 조종**하고 감시목록을 읽는다.
- TLS 없이 `/auth/forest-login`이 나가면 **숲나들e 비밀번호가 인터넷을 평문으로** 건넌다.
  사이트 자체가 "클라이언트 암호화 없음"이라 **앱→백엔드 HTTPS가 유일한 방어선**이다.

**결정:**
1. **인그레스 = 아웃바운드 터널.** 집PC에 인바운드 포트를 열지 않고 Cloudflare Tunnel 또는
   Tailscale로 백엔드를 노출한다. TLS가 터널에 내장되고 포트포워딩·고정IP·DDNS·인증서 갱신이
   전부 불필요하다(홈 라우터 공격면 0).
2. **모든 `/v1/*`에 Bearer 토큰 필수.** 개인 MVP라도 무인증을 허용하지 않는다. 토큰은 앱 최초
   셋업 시 발급해 기기에 보관하고 `Authorization: Bearer <appToken>`로 전송한다.
3. **전 구간 TLS 강제.** `/auth/forest-login` 포함, 평문 fallback 없음.

**대안(기각):**
- **인바운드 포트포워딩 + DDNS + Let's Encrypt:** 홈 라우터에 포트를 열어 공격면이 생기고
  인증서 갱신 운영부담 → 기각.
- **VPN(WireGuard) 상시 연결만:** 앱이 항상 VPN을 요구해 UX 부담, 핸드오프 순간 지연 →
  Tailscale(WireGuard 기반이나 매끈한 디바이스 메시)로 흡수.

**결과:**
- API계약 §0의 "무인증 허용" 문구를 삭제하고 Bearer 토큰을 기본으로 승격한다.
- Phase 0 작업분해에 **터널 셋업 + 앱 토큰 발급·검증** 태스크를 추가한다.
- [[0002-service-account-reads]]의 자격증명(숲나들e ID/PW)이 이 채널로만 오가므로 **채널 보안이
  자격증명 보안의 전제**다. 백엔드 저장 암호화(DPAPI)와 함께 이 ADR이 인증 모델의 반쪽을 이룬다.
