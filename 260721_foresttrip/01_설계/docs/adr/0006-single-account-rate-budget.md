# 단일 계정 처리량을 전역 레이트버짓으로 상한하고 커버리지를 1급 제약으로 노출한다

- 상태: 제안 (2026-07-22 아키텍처 리뷰에서 도출)

모든 폴링·조회는 **단일 숲나들e 계정 세션**을 공유한다. 방 단위 goods 조회는
로그인 + NetFunnel(~6–10s)에 예약(01)·대기(02) **2모드 전 페이지 순회**(§6-bis)까지 붙어,
대형 휴양림은 poll_target당 ~20–40초가 든다. 워커풀은 CONCURRENCY 6, 적응형 주기는 5–10분이며,
비밀번호 5회 오류 시 **계정 잠금** + 과호출 시 **소프트밴** 위험이 있다.

**문제:** 지금까지 "전역 레이트버짓 필수"라고만 하고 **수치도 상한도 없다.** 활성 감시가 예산을
넘으면 near-date urgent 감시가 normal을 굶기거나 poll_target이 제때 안 돌아 **취소표를 놓친다**.
이는 제품의 유일한 가치 제안이 **조용히 실패**하는 것이다. 용량이 정의되지 않으면 "감시를 몇 개까지
믿을 수 있는가"에 답할 수 없다.

**결정:**
1. **전역 레이트버짓 `R` req/min.** 보수적 값에서 시작해(작업분해 검증 "24h 무잠금" 통과 후 상향),
   스케줄러가 `poll_target.next_due_at` 순 + `priority`로 예산을 소비하고 초과분은 다음 창으로 이월한다.
2. **커버리지 상한 `N_max` 도출·노출.** `R`과 poll_target당 평균 비용에서 *신뢰 커버 가능한 활성
   poll_target 수*를 계산해 **1급 제약으로 문서화·모니터**한다. 활성 poll_target이 `N_max`에 근접하면
   앱에 "커버리지 저하"를 표면화하고 감시 추가를 경고/억제한다.
3. **백프레셔 정책 명시.** 예산 부족 시 urgent(예약 감시·D-day 근접)를 normal(대기 감시)보다 우선
   폴링한다. 굶는 target은 침묵시키지 말고 **상태로 노출**(앱에서 "폴링 지연 중" 표시).
4. **예산 절감책 — min-nofpr 병합.** poll_target을 인원(nofpr)별로 쪼개지 말고, 최소 nofpr로 1회
   폴링한 뒤 방의 `capacity`로 다운스트림 필터해 인원 변형을 **한 폴로 합친다**. 단 사이트 가용성
   로직이 순수 capacity 이상인지(§0.1 조사)를 확인한 뒤 적용한다. 성립 시 파편화된 예산을 크게 회복한다.
5. **자동재로그인은 실패 시 무재시도 + 백오프**(5회 잠금 회피) — 기존 규칙 유지, 실패 시 소유자 푸시.

**대안(기각/연기):**
- **계정 무제한 병렬:** 잠금·소프트밴 → 기각.
- **서비스계정 풀로 즉시 확장:** ToS 회색지대 + 복잡도 → Phase 4로 연기([[0002-service-account-reads]]).

**결과:**
- 단일 계정 상한상 **서비스계정 풀 전까지 멀티유저는 불가**함을 명문화한다(기획서 §1.3의 "멀티유저
  확장 대비"는 **스키마 대비**일 뿐 런타임 용량이 아니다).
- 커버리지 상한은 [[0003-tiered-freshness-snapshot]]의 스냅샷 커버리지와 직결된다 — 미감시 날짜의
  waitable을 warm하게 유지 못 하는 이유가 바로 이 예산 상한이다. 검색 계약의 신선도 분리(위험표 §9)와
  함께 읽어야 한다.
- 작업분해에 **N_max 산출 + 커버리지 표면화 + 백프레셔 우선순위** 태스크를 추가한다.
