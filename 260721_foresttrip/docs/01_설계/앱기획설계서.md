# 숲나들e 알림 앱 — 기획·설계서

> 특정일에 전국에서 예약/대기 가능한 자연휴양림·객실을 찾고, 관심 객실이 예약/대기 가능해지면
> 알려주는 안드로이드 앱과 그 백엔드. 코딩은 별도 세션에서 진행하며, 이 문서는 그 착수 기준이다.

- 버전: 1.0 (2026-07-22)
- 코드 위치: `260721_foresttrip/` (백엔드 소스는 `../src/`)
- 설계 문서(이 폴더 `260721_foresttrip/01_설계/`): `CONTEXT.md`(용어집), `docs/adr/0001~0006`, `API계약.md`, `DB스키마.md`, `작업분해_Phase0-1.md`
- 배경 문서(상위 폴더): `../인수인계.md`, `../코드설계서.md`, `../조사보고서.md`

## 1. 제품 개요

### 1.1 문제
숲나들e 인기 휴양림은 상시 만석이고, 취소표는 **분 단위로 나타났다 사라진다**. 사용자는
사이트를 수동으로 반복 조회해야 하며, 주말·공휴일은 추첨이라 "예약불가"의 이유조차 헷갈린다.

### 1.2 해결
1. **전국 특정일 조회** — 그 날 예약/대기 가능한 휴양림·객실을 한 화면에.
2. **감시(Watch) + 푸시 알림** — 관심 숙소상세(또는 휴양림 "아무 방이나")를 등록하면 백엔드가
   주기적으로 확인해 예약/대기 가능해지는 즉시 푸시.
3. **원탭 핸드오프** — 알림에서 숲나들e 공식 예약 페이지로 바로 이동해 사용자가 직접 확정.
   **앱은 예약을 대신 누르지 않는다**(자동예약·매크로 금지, ADR-0001).

### 1.3 범위
- **사용자 범위:** 개인(소유자 본인) 시작. 백엔드 데이터모델·API는 멀티유저 확장 대비 설계.
- **비범위:** 자동 예약·결제·매크로. 조회·알림·핸드오프까지만.

## 2. 핵심 도메인 제약 (설계를 지배하는 사실)

| 제약 | 함의 |
|---|---|
| **숙소상세 조회는 로그인 필수** (`fcfsRsrvtPssblGoodsDetls.do`) | 백엔드가 로그인 세션 보유·재사용, 자격증명 백엔드 암호화 저장(ADR-0002) |
| **NetFunnel 게이트** (예약가능 조회) | 순수 fetch 불가 → Playwright로 실제 페이지 로드 필요. 요청당 ~6–10초 |
| **"대기가능"은 방 단위 개념** (`fn_wtng_goSrch`, `rsrvtWtngSctin=02`) | 전국 대기 즉시조회 불가 → 스냅샷 필요(ADR-0003) |
| **대기신청은 D-2 마감** | 대기 감시는 D-2에 자동 종료 |
| **주말·공휴일 전날 = 추첨** | 선착순 조회에서 "예약불가"가 정상 → `hldtList`로 라벨링·안내 |
| **단일 서비스계정** | 폴링 전역 레이트버짓 필수(차단·계정잠금 방지) |
| **goods 목록은 모드별 페이지당 ~10개** (실측 확정) | 예약(01)·대기(02) 각 모드가 `fn_goPage`로 페이지네이션. 현 파서는 1페이지만 읽어 combined ~20에서 truncate → 방 많은 휴양림의 숙소상세 감시 누락. **전 페이지 순회 필수**(§6-bis) |

## 4. 기능 명세

### 4.1 전국 특정일 조회 (스냅샷 + 탭 실시간, ADR-0003)
- **1차(즉시):** 백엔드 스냅샷을 즉시 표시. 각 휴양림에 예약가능/대기가능 요약 + `snapshotAt`(신선도).
- **2차(온디맨드):** 휴양림 탭 → 그 휴양림만 실시간 재조회(수초) → 방 단위 상태·대기순위 확정.
- **필터/정렬:** 시/도, 신축(개관 N년 이내), 시설분류(숲속의집/휴양관…), 인원, 예약가능 우선.
- **추첨/선착순 라벨:** `hldtList` 기반으로 각 날짜에 `선착순/추첨/휴무` 배지 + 안내문.

### 4.2 감시(Watch)와 알림
- **대상 입도(Q6):** ① 숙소상세 지정(goodsId/라벨) ② 휴양림 "아무 방이나".
- **날짜:** 단일 날짜 또는 **희망기간**(박수 고정 + 기간 내 아무 체크인 + 요일 필터, Q13).
- **폴링(Q5):** 적응형 5–10분. 날짜 임박·취소 다발 시간대 가중. 전역 레이트버짓 상한.
  대기 전용 감시는 D-2 자동 종료, 예약 감시는 D-day까지.
  **폴링 단위는 (휴양림,날짜,인원)** — 같은 휴양림·날짜의 여러 방/휴양림감시는 조회 1회로 합침.
- **알림(Q7):** 2등급 —
  - `예약가능(즉시)` = **긴급**(소리·진동, DND 관통 옵션) — 최우선.
  - `대기가능` = **일반**(대기순위 포함).
  - 재알림 스로틀(계속 열려있으면 X분당 1회, 사라졌다 재등장 시 재알림). 히트 후 감시 유지.
  - DND 시간대 사용자 설정. 대기순위 개선(2→1순위) 알림은 옵션.
- **핸드오프:** 알림 탭 → 숲나들e 공식 예약 페이지 딥링크(§6 참조).

### 4.3 기본 차별화 (값싸고 확실 — 기본 탑재)
- 추첨/선착순 자동 구분 + 경고
- 대기신청 **D-2 카운트다운** 경보
- 신축·개관연도·시설 **필터/정렬**
- 가격·인원·면적·신축배지 요약 + 딥링크 **핸드오프 카드**

### 4.4 플래그십 차별화 (Q8, 전부 채택 — 단계적 투입)
| 기능 | 데이터 요구 | 단계 |
|---|---|---|
| 취소표 패턴 학습·명당시간 예측 | 스캔 시계열 이력(ADR-0004) | Phase 3 |
| 대기 성공 확률 추정 | 대기순위 궤적 이력 | Phase 3 |
| 유연 날짜 히트맵 | 희망기간 범위 스캔 | Phase 2 |
| 광역 조건 감시("강원 6인 숲속의집 아무거나") | 광역 폴링(레이트버짓 압박) | Phase 4(옵트인) |

> Phase 1부터 **시계열 이력은 계속 적재**한다(학습형 기능은 이력이 쌓인 뒤 켠다).

## 5. 인증·세션·자격증명 (ADR-0002)

1. 앱의 **숲나들e 로그인 화면**에서 소유자가 ID/PW 1회 입력 → `POST /auth/forest-login`.
2. 백엔드가 `login.js`로 로그인 → **세션(storageState)** 확보(= `auth.json` 상당) → 재사용.
3. 자격증명은 **백엔드에만 암호화 저장**(OS 시크릿/암호화 컬럼, 평문 로깅 금지).
4. **세션 만료 시 저장 자격증명으로 headless 자동 재로그인.** 단 **5회 오류 시 계정잠금**이므로
   실패 시 재시도 금지 → 백오프 후 소유자에게 "재로그인 필요" 푸시.
5. 앱은 숲나들e 비번을 저장하지 않고 **백엔드 세션토큰**만 보관.
6. 앱 사용자 식별(감시 소유자·푸시 대상)은 숲나들e 계정과 **별개 축** — MVP는 디바이스 등록 기반.

## 6. 딥링크 핸드오프 — ✅ 확정 (2026-07-22 실측)

**조사 결과(근거):**
- 숲나들e **공식 앱 존재**(안드로이드 `go.forest.foresttrip`, iOS `id1491335869`, 산림청). 단
  사실상 **WebView 래퍼**(iOS 1.1.3, 문서화된 URL scheme/associated domains 없음).
- **`https://www.foresttrip.go.kr/.well-known/assetlinks.json` → 404**. 즉 **Android App Links(검증된
  https 딥링크) 미선언** → https 링크 탭 시 공식 앱이 **자동으로 가로채 열지 않음**.
- 예약/goods 상세 페이지는 **NetFunnel 게이트** → 직접 GET 시 `com/alert.do`로 바운스 →
  **방/날짜 프리필된 딥링크는 원리상 불가**.
- **`fcfsRsrvtMain.do?hmpgId=FRIP&menuId=001001`** 는 로그인·게이트 없이 바로 열리는
  **일반예약(선착순) 검색 진입 페이지**. 단 **쿼리 파라미터 프리필은 미지원**(사용자가 시도/휴양림/
  날짜/인원 선택 후 '조회').

**확정 설계(폴백이 곧 정식 경로):**
1. **착지 타깃 = `https://www.foresttrip.go.kr/rep/or/fcfsRsrvtMain.do?hmpgId=FRIP&menuId=001001`**
   를 Android `ACTION_VIEW` 인텐트/Custom Tab으로 연다. 공식 앱이 설치돼 도메인 인텐트필터를
   등록했다면 "앱으로 열기" 선택창이 뜰 수 있으나 **검증/자동이 아니므로 의존하지 않는다**.
2. **프리필 불가 → "핸드오프 카드"가 검색값을 대신 채운다.** 알림/카드에 **휴양림명·시/도·입실~퇴실·
   인원·상태(예약가능/대기가능 N순위)** 를 크게 표기해, 사용자가 착지 페이지에서 **몇 탭이면 동일 검색**을
   재현하도록 한다. (편의: 날짜/휴양림명 클립보드 복사 버튼은 폴리시.)
3. **자동 입력/자동 예약은 하지 않는다**(ADR-0001). 임베디드 WebView에 값 주입하는 방식은 자동화로
   번지고 사이트 개편에 취약 → 채택하지 않음.
4. **추첨일 예외:** 대상 날짜가 추첨일이면 선착순 메인 대신 **추첨 예약 메뉴**로 안내(카드에 "추첨일"
   배지 + 해당 메뉴 링크). 우리 감시는 선착순/대기 대상이므로 일반 케이스는 위 1로 충분.

**미의존/후속(폴백에 영향 없음):** 공식 앱의 비검증 커스텀 스킴 존재 여부는 APK 리버싱이 필요하고
개편에 취약 → **의존하지 않음**. Phase 1 기기 테스트에서 "https 링크 탭 시 숲나들e 앱 선택창이
뜨는지"만 관찰해 UX 폴리시로 반영.

## 6-bis. goods 페이지네이션 — ✅ 확정 (2026-07-22 실측)

**조사 결과(근거):**
- 저장된 로그인 DOM `artifacts/goods-wtng.html`에 **페이징 위젯 실재**: `paging_count "(1/3)"` +
  `fn_goPage('2')`/`('3')`/마지막페이지. 즉 대관령 대기결과만 **3페이지**.
- 페이징은 내부 AJAX **`/rep/or/sssn/innerFcfsRsrvtPssblGoodsDetls.do`** 를 페이지 인덱스로 재호출.
- 현 `goods.extractRooms()`는 **로드된 1페이지만** 파싱 → 나머지 페이지 방은 누락.
- `artifacts/rooms-region.json` 실측 분포(133개 휴양림, 방 1376): **정확히 20개에서 잘린 휴양림 다수**
  (20 초과 0곳, 15+ 25곳). 무관한 휴양림들이 20에 겹치는 건 우연 불가 → **상한의 정체 = 예약 1페이지
  (~10) + 대기 1페이지(~10)만 읽은 combined 20**. 페이지네이션 미처리가 원인으로 확증.
- 부수: goods 페이지의 `captchaImg`는 **대기신청(apply) 단계용**. 우리는 조회만 하고 신청은 핸드오프
  이므로 무영향(ADR-0001 강화).

**확정 처리(Phase 1):**
1. **전 페이지 순회(권장):** 각 모드(예약 01 / 대기 02) 진입 후 `.paging_count` "(cur/total)"에서
   total(N)을 읽고, `fn_goPage('2')…('N')`를 Playwright로 구동하며 페이지마다 `extractRooms()`를
   호출·누적, `goodsId`로 dedup. 정확성이 속도보다 중요(감시 신뢰성) → 검증된 브라우저 흐름 재사용.
2. **비용/레이트버짓 반영:** 대부분 1페이지(74곳 ≤10방, 20곳 ≤5방)라 추가비용 0. 2–3페이지 필요한
   대형 휴양림은 ~25곳뿐 → 폴링 단위당 페이지수를 레이트버짓에 산입.
3. **대안(후속 최적화):** `innerFcfsRsrvtPssblGoodsDetls.do`에 페이지 파라미터를 직접 POST(세션/
   netfunnel/_csrf 헤더 재현). 빠르나 취약 → Phase 1은 1)로 확정, 성능 이슈 시 2)로 교체 검토.

**효과:** 숙소상세 감시 누락 제거 + 전국 스냅샷 방 수 정확도 향상(현재 대형 휴양림 과소집계 해소).

### 6-bis-1. 구현 시 실측된 함정 (2026-07-23 라이브 검증)

순회를 실제로 붙여보니 **DOM만 보고는 못 잡는 함정이 둘** 있었다. 재구현 시 반드시 지킬 것.

1. **모드 전환이 페이지 번호를 물려받는다 — 가장 위험.**
   예약 모드에서 2페이지를 보던 중 `fn_wtng_goSrch()` 로 대기 모드에 들어가면, 대기 결과도
   **`(2/5)` 로 시작**한다. "전환하면 1페이지"라고 가정하면 그 페이지를 통째로 놓치고, 심지어
   "2페이지 앵커가 없다"며 순회가 실패한다(현재 페이지는 `<strong>`이라 `<a>`가 아니다).
   → **`.paging_count` 에서 `cur` 와 `total` 을 함께 읽고, `cur` 를 뺀 나머지 전 페이지를 방문**한다.

2. **`fn_goPage(n)` 을 `evaluate` 로 직접 부르면 조용히 무시될 수 있다.**
   전역 함수이긴 하나 모드 전환 직후엔 위젯 준비 전에 발사돼 아무 일도 일어나지 않는다.
   → 사이트가 실제로 쓰는 경로인 **페이지 앵커의 inline `onclick` 을 클릭**한다. 앵커의 존재
   자체가 "준비 완료" 신호 역할도 한다. 전환 완료는 **목록 지문(goodsId 나열) 변화**로 판정한다.

**실측 효과 (대관령 0111, 2026-08-13, 2인):**

| 순회 없음(기존 PoC) | 20 (truncate) | — | — |
| 순회 추가, 함정 미해결 | 27 | 3 | false |

기존 PoC 데이터가 대형 휴양림에서 얼마나 과소집계였는지를 보여준다(20 → 66, 3.3배).
대기 물량이 특히 심하게 잘려 있었다(대기 49개 중 10개만 보였다).
